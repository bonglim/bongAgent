# KB Comrade · Bong에이전트 기능 명세서

| 항목 | 내용 |
|---|---|
| 문서 버전 | 1.0 |
| 작성 기준일 | 2026-07-15 |
| 대상 버전 | MVP 0.1.0 |
| 문서 상태 | 현행 구현 기준(As-Is) |
| 대상 사용자 | 은행 내부 업무 담당자 |

## 1. 목적

KB Comrade는 은행 직원이 개인 업무, 사내쪽지, 사후관리 고객과 일정을 한 화면에서 관리하고 자연어 명령으로 업무를 처리할 수 있게 하는 AI 업무 비서다. 본 문서는 MVP가 제공해야 하는 화면 동작, 기능 요구사항, 데이터 규칙, 예외 처리와 수용 기준을 정의한다.

## 2. 범위

### 2.1 포함 범위

- 상태별 ToDo Kanban 조회와 CRUD
- Drag & Drop을 이용한 ToDo 상태 변경
- 사내쪽지와 사후관리 고객의 조회, 등록, 삭제, 우선순위 변경
- 사내쪽지와 사후관리 고객의 ToDo 전환
- 자연어 기반 업무 조회·등록·수정·삭제
- 날짜 기준 또는 AI 추천 기반 우선순위 재조정
- LLM 모델 선택과 일반 질의응답
- 업무판 전체 상태의 Undo, Redo, 특정 시점 복원
- 캘린더, Agent API 정보 및 직접 호출 UI
- 데스크톱·태블릿·모바일 반응형 화면

### 2.2 제외 범위

- 사용자 로그인, 인증, 역할별 권한 관리
- 다중 사용자 간 실시간 협업과 데이터 동기화
- 실제 사내쪽지, 고객 CRM, 그룹웨어 연동
- 알림 발송, 파일 첨부, 반복 일정
- 데이터베이스 기반 저장, 다중 서버 프로세스 동시 쓰기
- 영구 감사 로그와 운영 수준의 개인정보·비밀정보 보호

## 3. 사용자와 사용 환경

### 3.1 사용자 역할

MVP의 사용자 역할은 `업무 담당자` 한 종류다. 사용자는 모든 ToDo, 사내쪽지, 사후관리 고객과 변경 이력을 조회하고 변경할 수 있다.

### 3.2 선행 조건

- 프론트엔드가 FastAPI 서버에 접속할 수 있어야 한다.
- 외부 LLM을 사용할 경우 선택 모델에 해당하는 API key가 설정되어 있어야 한다.
- API key가 없거나 외부 모델 응답이 유효하지 않아도 mock 또는 규칙 기반 fallback으로 핵심 업무 기능은 계속 동작해야 한다.

## 4. 화면 구성

| 화면/영역 | 주요 정보 | 주요 동작 |
|---|---|---|
| 상단 헤더 | 서비스명, 사용자 정보, 오늘 날짜 | 전체 새로고침, 설정 열기 |
| 좌측 메뉴 | 대시보드 메뉴와 업무 건수 | 접기/펼치기, 캘린더·쪽지·고객·설정 열기 |
| 오늘의 할일 | 할일·진행중·완료 Kanban | 생성, 상세/수정, 삭제, 상태 이동, 우선순위 변경 |
| 우선순위 원천 | 사내쪽지, 사후관리 고객 | 등록, 상세, 삭제, 우선순위 변경, ToDo 전환, 고객 AI 추천 |
| AI 채팅 | 대화 이력, 모델 선택, 입력창 | 자연어 명령 및 일반 질문 전송, 채팅 숨김/표시 |
| 캘린더 모달 | 현재 월의 날짜별 ToDo | 날짜별 업무 건수와 항목 확인 |
| 최근 기억 모달 | Undo/Redo 이력 | 되돌리기, 다시 실행, 특정 시점 복원 |
| Agent API 모달 | Agent별 endpoint와 설명 | URL 복사, 샘플 자연어 직접 호출, 결과 확인 |

## 5. 기능 요구사항

### 5.1 대시보드 공통

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-COM-001 | 초기 데이터 조회 | 앱 진입 시 ToDo, 사내쪽지, 사후관리 고객, 히스토리를 조회해 화면에 표시한다. |
| FR-COM-002 | 새로고침 | 새로고침을 실행하면 업무 데이터와 LLM 모델 목록을 다시 조회한다. 처리 중에는 중복 실행을 막고 로딩 상태를 표시한다. |
| FR-COM-003 | 사용자 피드백 | 생성·수정·삭제·조회 실패 또는 성공 결과를 화면 알림으로 표시한다. |
| FR-COM-004 | 반응형 레이아웃 | 화면 폭에 따라 메뉴, Kanban, 우선순위 패널과 채팅을 재배치해 주요 기능을 사용할 수 있어야 한다. |
| FR-COM-005 | 패널 표시 제어 | 사용자는 좌측 메뉴를 접거나 펼치고 AI 채팅 패널을 숨기거나 다시 표시할 수 있다. |

### 5.2 ToDo 관리

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-TODO-001 | 상태별 목록 조회 | ToDo를 `할일(todo)`, `진행중(doing)`, `완료(done)` 열로 구분해 표시한다. 각 열에 건수를 표시한다. |
| FR-TODO-002 | 정렬 | 우선순위는 `high → medium → low`, 같은 우선순위는 생성 시각 순으로 정렬한다. |
| FR-TODO-003 | 수동 생성 | 제목, 설명, 상태, 우선순위, 기한을 입력해 ToDo를 생성한다. 제목은 1자 이상이어야 하며 원천은 `manual`로 저장한다. |
| FR-TODO-004 | 상세 및 수정 | 카드를 선택하면 상세 모달을 열고 제목, 설명, 상태, 우선순위, 기한을 수정할 수 있다. |
| FR-TODO-005 | 삭제 | 카드 또는 상세 모달에서 삭제할 수 있다. 삭제 전 확인창을 표시하며 취소 시 변경하지 않는다. |
| FR-TODO-006 | Drag & Drop 상태 변경 | 카드를 다른 상태 열에 놓으면 즉시 화면을 갱신하고 서버에 상태를 저장한다. 저장 실패 시 이전 상태로 복구한다. |
| FR-TODO-007 | 빠른 우선순위 변경 | 카드의 우선순위 버튼을 누를 때 `low → medium → high → low` 순으로 변경한다. |
| FR-TODO-008 | 원천 표시 | ToDo가 쪽지나 고객에서 생성된 경우 원천 종류와 연결 정보를 유지한다. 직접 입력과 AI 명령 생성도 원천을 구분한다. |
| FR-TODO-009 | 연결 우선순위 동기화 | 연결된 ToDo 또는 원천 레코드의 우선순위를 변경하면 상대 데이터의 우선순위도 동일하게 갱신한다. |

### 5.3 사내쪽지 관리

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-MSG-001 | 목록 및 상세 조회 | 제목, 발신자, 수신일, 본문, 상태, 우선순위를 표시하고 상세 모달에서 전체 내용을 확인할 수 있다. |
| FR-MSG-002 | 등록 | 제목, 발신자, 수신일, 우선순위, 본문을 입력해 등록한다. 제목과 발신자는 1자 이상이어야 하며 최초 상태는 `unread`다. |
| FR-MSG-003 | 삭제 | 삭제 전 확인창을 표시한다. 연결된 ToDo가 있으면 ToDo 자체는 유지하되 쪽지 원천 연결을 해제한다. |
| FR-MSG-004 | 수동 우선순위 변경 | 우선순위를 순환 변경한다. 수동 변경 시 기존 AI 상세 순위와 선정 사유를 초기화한다. |
| FR-MSG-005 | ToDo 전환 | 쪽지 제목·본문·수신일·우선순위로 ToDo를 생성하고 쪽지 상태를 `todo_linked`로 변경한다. |
| FR-MSG-006 | 중복 전환 방지 | 이미 유효한 ToDo가 연결된 쪽지를 다시 전환하면 새 ToDo를 만들지 않고 기존 ToDo를 반환한다. |
| FR-MSG-007 | 우선순위 재조정 | 자연어 채팅을 통해 전체 쪽지에 중복 없는 1~N 순위, 등급과 한국어 사유를 저장한다. |

### 5.4 사후관리 고객 관리

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-CUS-001 | 목록 및 상세 조회 | 고객명, 관리 사유, 권고 조치, 예정일, 상세, 우선순위를 표시한다. |
| FR-CUS-002 | 등록 | 고객명, 관리 사유, 권고 조치, 예정일, 우선순위, 상세를 입력해 등록한다. 고객명과 관리 사유는 1자 이상이어야 한다. |
| FR-CUS-003 | 삭제 | 삭제 전 확인창을 표시한다. 연결된 ToDo가 있으면 ToDo 자체는 유지하되 고객 원천 연결을 해제한다. |
| FR-CUS-004 | 수동 우선순위 변경 | 우선순위를 순환 변경한다. 수동 변경 시 기존 AI 상세 순위와 선정 사유를 초기화한다. |
| FR-CUS-005 | ToDo 전환 | 고객명과 권고 조치로 제목을, 관리 사유와 상세로 설명을 구성하고 예정일·우선순위를 반영해 ToDo를 생성한다. |
| FR-CUS-006 | 중복 전환 방지 | 이미 유효한 ToDo가 연결된 고객을 다시 전환하면 새 ToDo를 만들지 않고 기존 ToDo를 반환한다. |
| FR-CUS-007 | AI 우선순위 추천 | 고객 패널에서 선택 모델로 전체 고객의 우선순위를 추천받아 저장하고 요약과 고객별 선정 사유를 모달에 표시한다. |
| FR-CUS-008 | 자연어 재조정 | 채팅에서 전체 고객에 중복 없는 1~N 순위, 등급과 한국어 사유를 저장하고 연결 ToDo의 등급도 동기화한다. |

### 5.5 자연어 Agent 및 LLM

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-AI-001 | 모델 목록 조회 | 서버 설정에서 공개 가능한 모델 id, 표시명, provider와 기본 모델을 받아 선택기에 표시한다. API key와 환경변수 이름은 노출하지 않는다. |
| FR-AI-002 | 모델 선택 | 사용자가 선택한 모델 id를 채팅, AI 추천, Agent 직접 호출 요청에 포함한다. 목록 조회 실패 시 프론트 fallback 모델을 사용한다. |
| FR-AI-003 | 의미 기반 라우팅 | 입력을 `message`, `customer`, `todo`, `general` 중 하나로 분류해 해당 Agent에 전달한다. LLM 분류 실패 시 키워드 규칙으로 라우팅한다. |
| FR-AI-004 | ToDo 자연어 명령 | ToDo 등록, 조회, 상태 변경, 삭제 명령을 처리하고 구조화된 intent와 한국어 답변을 반환한다. |
| FR-AI-005 | 쪽지 자연어 명령 | 쪽지 등록, 조회, 삭제, 우선순위 재조정과 저장 데이터 기반 질문을 처리한다. |
| FR-AI-006 | 고객 자연어 명령 | 고객 등록, 조회, 삭제, 우선순위 재조정과 저장 데이터 기반 질문을 처리한다. |
| FR-AI-007 | 일반 질의응답 | 업무 데이터 명령이 아닌 질문은 선택한 LLM provider에 전달한다. |
| FR-AI-008 | 변경 화면 동기화 | Agent 응답이 데이터를 변경한 intent이면 관련 목록과 히스토리를 다시 조회해 화면을 갱신한다. |
| FR-AI-009 | 안전한 확인 요구 | 삭제·수정 대상이 불명확하거나 일치 항목을 찾지 못한 경우 임의 변경하지 않고 `needs_confirmation` 또는 안내 답변을 반환한다. |
| FR-AI-010 | Provider fallback | API key 부재, 호출 실패, 잘못된 구조화 응답이 발생해도 mock 또는 규칙 기반 결과를 반환한다. |

지원하는 주요 intent는 다음과 같다.

| 도메인 | intent |
|---|---|
| ToDo | `create_todo`, `update_todo`, `delete_todo`, `query_todos` |
| 사내쪽지 | `create_message`, `delete_message`, `query_messages`, `set_message_priorities` |
| 사후관리 고객 | `create_customer`, `delete_customer`, `query_customers`, `set_customer_priorities` |
| 공통 | `needs_confirmation`, `chat` |

### 5.6 우선순위 재조정

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-PRI-001 | 명시적 날짜 정렬 | 사용자가 최근/오래된 날짜 또는 빠른/늦은 예정일을 명시하면 LLM을 호출하지 않고 저장 날짜로 결정적으로 정렬한다. |
| FR-PRI-002 | AI 추천 정렬 | 정렬 기준이 없는 재조정 요청은 긴급성, 준법·보안, 금융·고객 영향, 기한, 후속 조치 필요성을 종합 평가한다. |
| FR-PRI-003 | 결과 완전성 | 모든 대상에 `priority`, 중복 없는 1~N `rank`, 한국어 `reason`이 있어야 한다. |
| FR-PRI-004 | 구조 검증과 fallback | 항목 누락, 알 수 없는 id, 중복 rank, 허용되지 않은 등급, JSON 파싱 실패가 있으면 전체 결과를 규칙 기반으로 다시 생성한다. |
| FR-PRI-005 | 등급 구간화 | 정렬된 전체 항목을 상·중·하 구간으로 나눠 `high`, `medium`, `low`를 부여하고 정확한 전체 순서는 `priority_rank`에 저장한다. |
| FR-PRI-006 | 표시 순서 | `priority_rank`가 있는 항목은 작은 값부터, 없는 항목은 등급 순으로 표시한다. |
| FR-PRI-007 | 원자적 이력 | 전체 재조정은 하나의 히스토리 단위로 저장해 한 번의 Undo로 복원할 수 있어야 한다. |

### 5.7 Undo, Redo 및 시점 복원

| ID | 요구사항 | 상세 동작 및 수용 기준 |
|---|---|---|
| FR-HIS-001 | 변경 전 스냅샷 | 쓰기 작업 전에 ToDo, 쪽지, 고객 전체 상태를 하나의 스냅샷으로 저장한다. |
| FR-HIS-002 | Undo | 가장 최근 변경 전 상태를 복원하고 복원 직전 상태를 Redo 스택에 저장한다. |
| FR-HIS-003 | Redo | 가장 최근 Undo 이전 상태를 다시 적용하고 현재 상태를 Undo 스택에 저장한다. |
| FR-HIS-004 | 특정 시점 복원 | 사용자가 선택한 히스토리 id의 스냅샷으로 복원한다. 복원 후 Redo 스택은 초기화한다. |
| FR-HIS-005 | 버튼 상태 | 실행할 이력이 없으면 Undo 또는 Redo 버튼을 비활성화한다. |
| FR-HIS-006 | 보관 개수 | Undo와 Redo 이력은 각각 최근 최대 30건을 유지한다. |

## 6. 업무 규칙

### 6.1 상태 및 우선순위

- ToDo 상태는 `todo`, `doing`, `done`만 허용한다.
- 우선순위 등급은 `high`, `medium`, `low`만 허용한다.
- 빠른 우선순위 변경 순서는 `low → medium → high → low`다.
- 수동 우선순위 변경은 기존 `priority_rank`를 `null`, `priority_reason`을 빈 문자열로 초기화한다.

### 6.2 연결 데이터

- 쪽지 또는 고객에서 ToDo를 만들면 양쪽 레코드에 연결 id를 기록한다.
- 원천 레코드를 삭제하면 연결 ToDo는 삭제하지 않고 `linked_type`, `linked_id`만 해제한다.
- 연결 ToDo를 삭제하면 원천 레코드의 `linked_todo_id`를 해제한다.
- 연결된 양쪽 데이터의 우선순위 등급은 동일하게 유지한다.

### 6.3 저장과 동시성

- 데이터는 JSON 파일에 저장한다.
- 프로세스 내부 쓰기는 `RLock`으로 직렬화한다.
- 여러 서버 프로세스가 같은 파일에 동시에 쓰는 구성은 지원하지 않는다.

## 9. 오류 및 예외 처리

| 상황 | 기대 동작 |
|---|---|
| 필수 입력 누락 또는 enum 오류 | HTTP 422를 반환하고 저장하지 않는다. |
| 존재하지 않는 리소스 조회·수정·삭제 | HTTP 404와 한국어 오류 상세를 반환한다. |
| Drag & Drop 또는 빠른 우선순위 저장 실패 | 낙관적으로 변경한 화면 상태를 직전 상태로 되돌리고 오류를 표시한다. |
| 대시보드 일부/전체 조회 실패 | 오류 메시지를 표시하고 앱이 비정상 종료되지 않게 한다. |
| LLM 모델 목록 조회 실패 | 프론트 fallback 모델 목록과 기본 모델을 사용한다. |
| 외부 LLM 호출 실패 | mock 또는 규칙 기반 fallback으로 처리하고 가능한 경우 fallback 사용 여부를 결과에 포함한다. |
| 유효하지 않은 우선순위 추천 결과 | 일부만 저장하지 않고 전체 규칙 기반 결과로 대체한다. |
| Undo/Redo 대상 없음 | 데이터는 변경하지 않고 실행할 변경이 없다는 메시지를 반환한다. |

## 11. 핵심 인수 테스트 시나리오

### AT-01 ToDo 생성 및 상태 이동

1. 사용자가 제목이 있는 ToDo를 생성한다.
2. 새 카드가 지정한 상태 열에 표시된다.
3. 카드를 `진행중` 열로 Drag & Drop한다.
4. 새로고침 후에도 상태가 `doing`으로 유지된다.
5. Undo를 실행하면 생성 또는 상태 변경 직전 스냅샷으로 복원된다.

### AT-02 사내쪽지 ToDo 전환

1. 사용자가 제목과 발신자를 입력해 쪽지를 등록한다.
2. 쪽지의 `ToDo 추가`를 실행한다.
3. 쪽지 내용과 우선순위를 반영한 ToDo가 한 건 생성되고 쪽지는 연결 상태가 된다.
4. 같은 쪽지를 다시 전환해도 ToDo 수가 증가하지 않는다.
5. 연결 ToDo의 우선순위를 바꾸면 쪽지 등급도 동일하게 변경된다.

### AT-03 고객 AI 우선순위 추천

1. 사후관리 고객이 2건 이상 등록되어 있다.
2. 사용자가 모델을 선택하고 AI 추천을 실행한다.
3. 모든 고객에 중복 없는 1~N 순위, 허용된 등급, 한국어 사유가 저장된다.
4. 결과 모달에 전체 요약과 고객별 선정 사유가 표시된다.
5. 연결 ToDo의 우선순위가 고객 등급과 동기화된다.
6. Undo 한 번으로 추천 직전 전체 상태가 복원된다.

### AT-04 명시적 날짜 재조정

1. 서로 다른 관리 예정일을 가진 고객을 준비한다.
2. 채팅에 `사후관리 고객을 최근날짜 순으로 우선순위를 지정해 주세요`를 입력한다.
3. 고객이 예정일 내림차순의 1~N 순위로 표시된다.
4. 답변에 적용 기준과 항목별 사유가 포함된다.
5. 외부 LLM API key가 없어도 같은 정렬 결과를 얻는다.

### AT-05 오류 복구

1. API 서버 연결을 끊은 상태에서 카드 상태 이동을 시도한다.
2. 화면의 카드가 원래 상태로 돌아온다.
3. 사용자에게 저장 실패 메시지가 표시된다.
4. 앱의 다른 화면 요소는 계속 조작할 수 있다.
